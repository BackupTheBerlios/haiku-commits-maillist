2008/3/21, Ingo Weinhold <ingo_weinhold at gmx.de>:
>
>  On 2008-03-21 at 17:56:27 [+0100], Bruno Albuquerque <bga at bug-br.org.br>
>  wrote:
>
> > bonefish at BerliOS escreveu:
>  >
>  > > Haiku patch for libtool 1.5.26. The test suite succeeds save for test
>  > > link-order.test, which seems to require ld support for undefined symbols
>  > > in libraries. We'll review that after bootstrapping Haiku native
>  > > binutils and gcc.
>  >
>  > Not sure if this matters in this context or not, but under BeOS the
>  > trick to get shared libraries generated by libtool in most open source
>  > programs was to pass -no-undefined to it.
>
>
> Yep, that helps unless the library really uses undefined symbols. The best
>  strategy will be to reconfigure ld so that it allows undefined symbols,
>  though. Our runtime loader did support that already, anyway; it's currently
>  just disabled due to certain issues with R5 binaries. After adding support
>  to detect the build tools version at runtime, we should be able to reenable
>  it and use it on a per executable basis.

This reminds me. One problem I kept having was when I used my linux
machine with a GCC 4.2 crosscompiler to compile a native haiku GCC, it
failed because of linking problems. I cannot remember them right now,
but in order to solve it, I had to pass '-z defs' to gcc. I don't know
if this is a lead or not, but I hope it helps.
