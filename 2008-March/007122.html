That looks kind of inconvenient to use. Wouldn't it be nicer to use it like 
"filter <pattern> <command> <args> ...", so that it applies only to the 
given command? There's already a flag add_debugger_command_etc() flag that 
tells the command line parser not to parse the arguments, but rather pass 
them as single string to the command. "filter" would then only have to 
extract the pattern argument (the command line parser could provide a new 
function to do that), set the pattern, run the program, and unset the 
pattern.

BTW, I originally considered adding a simple pipe feature to the language, 
which would look pretty much like an ordinary shell pipe (e.g. "threads | 
grep Looper"), but work a little differently: The second command would be 
invoked whenever the first command prints a line, with that line as 
additional parameter. The output of the first program would not appear on 
screen, of course, unless the second command reprints it. I couldn't think 
of much uses for this feature other than "grep", though.

CU, Ingo
