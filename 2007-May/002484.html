On 2007-05-29 at 19:16:59 [+0200], Niels Reedijk <niels.reedijk at gmail.com> 
wrote:
> 2007/5/29, Ingo Weinhold <bonefish at cs.tu-berlin.de>:
> 
> > > How about:
> > >
> > > \typedef typedef dev_t mount_id
> > > \brief A \c mount_id refers to a specific mounted volume, whether it
> > > is on an actual disk or a virtual volume (such as image files).
> >
> > Well, I find "virtual volume" not adequate. Particularly not when 
> > referring
> > to a mounted image file. I could understand when you refer to FS systems
> > like NFS, CIFS, RAM FS, or Google FS as virtual, since they don't
> > (directly/locally) use an underlying storage medium -- on the other hand,
> > they aren't any less concrete when it comes down to storing data on them.
> > Technically there's almost no difference between mounting a partition and 
> > an
> > image file, though. I'd either drop the distinction (i.e. the ", whether
> > ..." part) or discriminate between disk-based (BFS, UDF,...) and non
> > disk-based (NFS, RAM FS, Google FS) file systems -- maybe using a better
> > term for "disk-based", although I don't find it that bad at all.
> 
> Let's put it this way, I think that any developer who will write file
> system modules understands that file systems can both exist on disk
> and can be from an image, or completely virtual.
> 
> So let's focus on the fact that mount_t refers to an instance.
> 
> \typedef typedef dev_t mount_id
> \brief A \c mount_id refers to a specific instance of a mount request.
> 
> Only the term mount request should be improved (could you replace it
> with file system instance? but then it makes no sense in that it is
> also an argument to mount()).

I would simply stick with "mounted volume". The reason that it is passed to 
the mount() hook is, that it is assigned by the VFS and the FS has to use it 
(in mount() already). With the exception of the mount() hook, where the 
volume is about to be mounted (most of the VFS volume related initialization 
is done already), the ID indeed always refers to a mounted volume.

CU, Ingo
