wrote:
> "François Revol" <revol at free.fr> wrote:
> > > * Rewrote terminal ID code to use a temporary file - this fixes bug
> > > #1174.
> > I don't like this.
> > This looks highly Unixish and for once I don't find this a good
> > thing.
> > There is no reason to have to write to disk for just opening a
> > Terminal.
> > It will fail on read-only media (install CD). Ok we can mount a
> > ramdisk
> > for /tmp but it's not done yet, and it's wrong anyway.
> > Plus if the fs is screwed up or locked up it will kill one's last
> > chance to open a terminal to do whatever could unblock the situation
> > because it will just lockup.
> 
> Feel free to suggest a cleaner method :-)
> There is just no other clean way to have a global and shared resource
> that is not accidently created twice (ie. there is no B_UNIQUE_NAME
> flag yet - then, we could use a cloned area).
> The only other ugly thing would be to create a port and use
> set_port_owner() to let it be owned by the system team - that's how
> BeOS does it. Under Haiku, this won't be allowed, though.

We could add atomicity support to BClipboard, i.e. a BClipboard::Commit() 
version that would fail, when the clipboard did already exist or was 
changed by someone else in the meantime.

When I wrote the UserlandFSServer I mistook BClipboard::Lock()/Unlock() for 
locking the clipboard system-wide and used it for some similar 
inter-application synchronization. The code is still in there and waits to 
be replaced. So I'd be glad, if we had a good solution.

CU, Ingo
