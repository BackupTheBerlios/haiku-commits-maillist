"Hugo Santos" <hugosantos at gmail.com> wrote:
>   It's also important to understand in which places a lack of memory
> would be fatal. I might be missing something, but the only situation
> where to me this would be critical would be in the allocator itself,
> in a situation where you would need memory to allocate memory. As
> hashing is only used for large blocks of memory (i.e. >= 512 bytes)
> this is very uncritical.

Yes, you're missing something indeed: there are lots of places in the 
current code that need to run with interrupts turned off (which 
basically means no allocations) but must still be able to insert 
objects into a hash.
Furthermore, allocating memory can always fail - inserting into an open 
hash cannot by definition - plus it's always guaranteed to be fast 
(which is also important when running with interrupts turned off).
