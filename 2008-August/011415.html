> Hmm. This problem could be solved more elegantly by using a  
> BMessageRunner. This way the polling thread could be avoided  
> completely. IIRC src/apps/pairs/PairsWindow.cpp uses BMessageRunner

Ok, i had a look. I think the case is a bit different though, i'll explain.
The first reason i did it that way is that i need a failry high 
resolution ~50Hz and i didn't want to flood the view with messages 
(although i'm sure it can handle it quite well). The second reason is 
that i wanted to make the solution pluggable with minimal code to add in 
the view using it, hidding the implementation, and also a solution that 
would be compatible if the input server provides this message someday. 
So from the class user point of view i find it pretty elegant, but that 
can be enhanced off course.

If someone shows me a quick way to write an implementation with a 
BMessageRunner while keeping the same modularity, then i'll reconsider 
it. I've considered this option seriously for my case, but i'm not sure 
it's simpler and/or more efficient.
Even better, if it's a small change go ahead and do it :) Honestly, 
unless there's a real bug/inefficiency in my approach,  i don't really 
plan on passing too much hours on this! Hope you understand.
