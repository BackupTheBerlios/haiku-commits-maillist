Hi,

> If I drag and drop a folder containing 5 files onto one containing 500,
> the last thing I want or expect is for the 500 files to be destroyed. Sorry,
> but if you want that behavior so badly, then you can either revert that patch
> in your private copy, or you can drag and drop that folder into the trash
> first, since that's the net effect you're asking for anyways.

And that is really the strongest argument pro the new behaviour. If Tracker did not offer it, it would be really, really tedious or nearly impossible for someone to do this otherwise/manually. The opposite, doing something like the old behaviour is really as simple as removing the target folder first. Compare the easiness of that (O(1)) to a manual merge (Puh!).

What I'd love though is more control over the behaviour - three options only:

What to do with duplicates?
1) Keep newer
2) Always replace
3) Always ask

In all scenarious, nothing would be done if the files "are the same" (size and modification time). It may be useful to force replacing, but that would be a another option, since it would only make sense if the user knows there may be some mixups he wants to avoid by using the copies from the source folder always. Note that "2b) Always keep" can be achieved by switching source and target and using "Always replace".
