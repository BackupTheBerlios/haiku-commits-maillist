> > Log:
> > Added a small test suite for the runtime loader. The tests are designed
> > to succeed on Linux. FreeBSD fails some tests since it seems to have a
> > different load order (depth-first instead of breadth-first). Haiku fails
> > a lot of tests due to its POSIX non-compliant dlopen().
> >
> 
> Pardon if this is a somewhat ignorant question, but what difference
> does depth-first vs breadth-first make in this context? Is it with
> respect to symbol searching / overriding in the case of things like
> weak aliases or.. ?

Yep, the load order is mainly relevant for resolving undefined symbols or 
looking symbols up in the global scope (dlsym(dlopen(NULL, 0), 
"symbolName")), and pretty much only when several images define the same 
symbol. I might have been mistaken with FreeBSD using a depth-first load 
order. It looks more like they just don't implement dlopen(NULL, 0) 
correctly.
