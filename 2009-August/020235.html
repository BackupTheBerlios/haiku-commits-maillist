wrote:
> Ingo Weinhold <ingo_weinhold at gmx.de> wrote:
[...]
> > I suppose we should introduce a userland memory barrier API for
> > that purpose.
> 
> There already is, at least more or less - just use the atomic_*()
> functions, and you should be fine.
> I am not sure we need anything beyond that, although it surely wouldn't
> hurt to add some.

The atomic_*() functions indeed create memory barriers on Intel 64/IA-32. 
This is merely by accident though, since they are implemented via locking 
instructions and those have the property of preventing reordering of memory 
accesses on the currently existing processor families of those 
architectures. Future Intel 64/IA-32 processor families might not have that 
property (I haven't read that this is a general architecture property that 
won't change at least), and I believe the atomic_*() implementations for 
other architectures (e.g. PPC) do not create memory barriers at all.
