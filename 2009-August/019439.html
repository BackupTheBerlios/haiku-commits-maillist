Hi Jonas,

Jonas Sundström wrote:
> An overlay filesystem is not the answer.
>
> An content2attribute overlay on e.g. an NTFS volume would
> have to inspect every mp3 file to have anything to show you
> when you make a query on the volume. It would have to inspect
> every JPEG and email and Ogg and PDF to be able to show
> the attributes you might want to see in Tracker for these
> filetypes. It would have horrible performance.
>
> A filename-extension-to-filetype-attribute overlay, e.g.
> on NTFS could translate (and hide?) filename extensions
> and present them as filetype attributes, but little would
> be gained from that. Haiku likely has no problem understanding
> the filetypes of the files you want to interact with on
> a foreign filesystem.
>
> The attribute overlay we have for iso9660 CDs and could
> use for FAT usb sticks doesn't translate attributes to
> a form that other systems understand and it doesn't 
> alter file content. The -filesystem- can't embedd 
> attribute data in its files and have other systems 
> understand it. Filesystems are generally content-unaware,
> passive storage.
>
> Tracker can be extended to do all kinds of special
> case things when copying/moving between volumes with
> different properties, but it gets messy real fast
> with a lot of guesswork about what filetypes' extensions
> are desired on what filesystems used by what operating
> systems, and by what kind of user. (I fear a dialog 
> worse than Tracker.NewFS and an ever-growing preference
> panel.)
>   

I was not talking about an overlay system to allow live queries on 
non-BFS volumes (although I would not mind that capability). What I 
meant was a system that would retain the ability to use filenames 
without extensions in Haiku, with the OS itself adding the extensions 
when those files are copied/moved to a non-BFS volume (or viceversa).

How to implement this cleanly and without overdoing it, I don't have a 
clue. That's for the engineers to figure out. :)

Cheers,

Jorge/aka Koki
