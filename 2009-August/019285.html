Hmmm, having a standard like that might be good, but I've personally
just been playing it by ear. In other words I find what needs to be
tabbed out the most and then match everything else up. If there are no
virtual or static keywords I don't worry about tabbing out to account
for them. This works, but it certainly isn't consistent. I don't know
if this is what was meant in the guidelines either (having the example
code in the guidelines is good, but it doesn't cover all cases.)

Having a set number of tabs would also be easier to automate with
tools. I wrote a small Ruby script for renaming private methods with
starting underscores (which I used on some ShowImage code), and I was
thinking of expanding this to other tools, especially one for fixing
up these headers, since it is tedious. It wouldn't be hard to have
tools at least to check for coding violations, but fixing them all is
harder.

Also as I've mentioned before there is a tool they added in WebKit for
checking style violations too. It is written in Python, though I'm
more of Ruby guy myself. But it might be easier to adapt than writing
our own (unfortunately their guidelines differ from ours, but not too
much.)
