> I've run into the same thing and thought about it a bit. Generally the
> message format is just more handy on the application side of things,
> while the text format is easier to hand edit. A solution to this problem
> would be to devise a text based flat format. Dano introduced a way to
> specify what flat version you intend to store a BMessage in as part of
> the switch to the new message format. The same API could be added to
> Haiku. With that you could specify that you want it flattened to a
> textfile. It'd just use a magic string at the beginning of the file so
> the unflatten hook would know how to handle it. This would make it easy
> for applications to get the best of both worlds without the overhead of
> having to parse text based settings files. If we can agree on adding
> such functionality I'd be willing to implement it as I've already played
> around with that.

Please don't add the functionality to BMessage directly. That's really not 
nice from a design point of view, as it is mixing a data model with support 
for its various persistency strategies/formats. IMHO an elegant design 
would abstract the various containers for typed data (read: an 
interface/abstract class (BTypedDataContainer)) and provide persistency 
support using that interface (interface BTypedDataStore). This would give 
great flexibilty and allow easy conversion between the formats.

CU, Ingo
