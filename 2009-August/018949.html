Hi Axel,

Playing with a virtual mode (workspace resolution > screen resolution works
very nice if there's hardware cursor support: if that's used the driver will
automatically pan and scroll around the workspace following the mouse.

Of course that could also be done with a soft cursor but then the app_Server
would need to call the accelerant giving it the new coordinates via some
(new?) hook.

I just tested with a (nvidia/matrox) card and dualhead setup: if you set a
dualhead (not clone) mode the workspace is twice as wide as the screen: the
Haiku preflet then does nolonger work (all modes greyed out).

The cursor doesn't have to be followed by the driver in such a mode since
it's always visible (left or right screen). Unless of course you'd set a
mode that has a virtual resolution above twice the screen: in that case
(with a hardcursor) the driver lets both screens follow the mousecursor.

I always tested this using a small command line tool I 'wrote' called
virtualscreen (wich btw works on haiku now). I could very nicely find out
the hardware granularity and max sizes for setting (virtual) screens with
that. It was kind of cool having a workspace of above 10.000 by 10.000
pixels ;-)

Good to know you're OK with the changes. Thanks!

Rudolf.
