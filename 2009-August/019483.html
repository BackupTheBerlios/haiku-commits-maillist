Well i understand what Jonas means, attributes are "meant" for meta-
data, that is data (or info) about the data contained in the file. 
Nothing prevents you to use them for storing data, but i see how it can 
be considered hackish to use it that way. It's an extreme use of a 
great feature (abuse?), and it shouldn't be imitated for every purpose.
There's the same problem with xml, where people often ask themselves if 
they should put information in attributes or in child nodes. It's mostly 
a semantical choice.

Similarly to what François noted about file extensions, having a 
possibility doesn't mean it's the only way. The possibility to store 
any data in attributes doesn't mean that zero-sized files is a good 
idea.

To get back on the initial subject; The file extension question is not 
something we can completely solve overnight. A file type is a meta-data 
and attributes are definitely the place for that, but it's a complex 
problem, involving interoperability, legacy etc... are we going to 
forbid .h .cpp .o .so .html .rdef filenames... Are we going to forbid 
users to name their files like they want?
