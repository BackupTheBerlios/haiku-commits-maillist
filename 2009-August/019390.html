> Actually there is a libattr in Linux which we might want to provide so
> it can be picked up by ported apps...

It seems kind of hard to find information on libattr. I spent a few minutes 
googling, but could even find out whether there's some kind of official 
home page or if it is a subproject of some other project. At any rate its 
headers seem to be installed in /usr/include/attr/. There's an xattr.h, 
which exposes the same interface as <sys/xattr.h>. objdump confirms that 
the library contains these syscalls, too (don't know why -- glibc provides 
them already). Furthermore there's attributes.h, which provides an 
"almost-IRIX-compatible extended attributes API", and libattr.h with two 
functions to copy files. Save for the latter the library seems utterly 
useless to me. If it is actually used, I'd consider it an excellent 
third-party opportunity to port it. :-)

> Now I'm not entirely sure this is the best mapping...

Why would we care how attribute namespaces are encoded in ext2? We just 
want to map the generic API.
