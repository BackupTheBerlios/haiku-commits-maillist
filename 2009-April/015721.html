mmlr at mail.berlios.de schrieb:

> * Indeed we need to return a status from the interrupt handler, as for PIO
>   transfers there is no way of knowning whether or not the interrupt at hand
>   was ours.

That is the reason why PIO transfers should be done with polling. You 
simply don't know if the interrupt that happend was done by the IDE 
controller, which IDE controller, etc.

Especially with interrupt sharing, not using interrupts is much saver.
I don't think there is a benefit in trying to do PIO with interrupts.
PIO transfers are rare anyway, so this is no performance impact.

DMA transfers are save to do with interrupt, as the controller can tell 
when it caused the interrupt.

The idea when doing the ata bus manager was to reduce complexity by 
avoiding PIO interrupt handling and tagged command queuing.
