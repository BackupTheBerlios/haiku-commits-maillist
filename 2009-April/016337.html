Hi,

>> The FreeBSD approach to read the surroundings (+/- 8 pages) on a page 
>> fault 
>> sounds better to me. Assuming a certain locality of related code the 
>> odds are 
>> mostly code that will indeed be needed is loaded.
> 
> That might work well enough on HDs that it will turn out equally fast 
> as the current prefetch method, although it will probably be slower 
> when starting apps.
> However, the current method is great for CDs, where on-demand paging 
> should be avoided.

I have not had a chance to do any performance measurements, but just 
because Haiku feels "fast enough" when booting from a HD should not mean 
the problem is not present in the exact same way on hard disks. To me it 
seems logical that improvements achieved for CD booting should reflect 
on HD booting with a similar factor, when dividing out number of heads, 
seek latency versus throughput latency and so on.

As I have told Axel already, it seems to me that CD booting would 
benefit from some kind of "block cache" that sits below the file cache. 
Maybe it could be done with another file system overlay (or rather 
underlay)?

Best regards,
-Stephan
