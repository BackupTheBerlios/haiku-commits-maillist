> Are you integrating PCI and ACPI with device manager, so that we
> only have one (shared) device tree? would be very nice. also,
> PCI bus manager should be a child of the ACPI bus manager, on
> ACPI enabled systems, and probably use ACPI methods for device
> enumeration, and more important, configuration of IRQs.

Well, one thing after the other :-)
I don't have a plan for ACPI yet - technically, it's optional, and PCI 
is not really a child of it. Plus, it exports its own set of devices.
I haven't really digged into ACPI yet, but I currently think that ACPI 
will live as a child of root side-by-side with PCI. ACPI would export a 
module for PCI that delivers the functionality the PCI bus manager may 
want to use. As I said, I'm not sure about this at all yet.

> Also, passing the BIOS e820 memory map to the ACPI or PCI bus
> manager would be useful to solve bugs 3 to 5.

That should certainly be possible. We could put more info into the 
kernel args (they already contain the physically usable memory), and 
eventually take the boot_item approach as VESA does.
