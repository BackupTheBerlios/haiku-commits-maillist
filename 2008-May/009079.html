François Revol wrote:
> > > This shows that with our default installation there are quite a few 
> > > "condensed" styles and also an "extra light" variation. The condensed 
> > > ones should probably get their own face flag. The "extra light" and 
> > > other 
> > > unhandled styles should probably not be classified as a 
> > > B_REGULAR_FACE as 
> > > this could lead applications to use them in a wrong way yielding 
> > > unexpected results (i.e. firefox that picks up the "extra light" 
> > > style 
> > > for it's interface font).
> > 
> > I am on it. Thanks for investigating! :-)
> 
> Hmm that's a nice idea, but instead of having multiple 
> B_(EXTRA_)[n]_LIGHT flags (you'd need the _HEAVY versions too anyway, and 
> it's an uint16!) I'd rather have B_SERIF and B_FANTASY or whatever do 
> describe non-regular-looking ones. B_SCRIPT maybe ? I rarely that face 
> thing, I try to always use family and style, like in XEmacs, which uses a 
> string descriptor including size anyway, but for NetSurf I had to map 
> flag-like constants for weight and others.
> 
> The weight thing seems to be more of a float value by itself on other 
> platforms though, like Pango and RiscOS.

Yes, I think it doesn't make too much sense how Be designed that. The flags 
could be used for stuff like B_STRIKE_THROUGH_FACE our B_UNDERLINE_FACE, 
but not for the actual face/style. While it makes sense to combine 
B_BOLD_FACE and B_ITALIC_FACE, it doesn't make sence that B_REGULAR_FACE is 
even part of the flags. And now there are even more face types, and they 
just can't be combined well. I think this API just doesn't map well to how 
font files work.

Probably can't do much about it until after R1 though.
