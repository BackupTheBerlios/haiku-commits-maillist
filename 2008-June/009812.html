You mean why you ended up replacing rubout (delete previous char) by delete 
functionality for character 0x7f in less' source code? That's the character 
Be's Terminal produces when the delete key is pressed (unlike xterm and 
others), and I guess you didn't like changing inputrc for some reason.

> And does that mean we don't have good defaults
> without the inputrc anymore?

I don't think we ever had good defaults. Besides, the whole terminal 
handling is pretty silly anyway. Neither does termcap specify capabilities 
for the complete key set nor are the termcap entries complete with respect 
to the existing capabilities (supposedly due to the stupid 2 KB limit per 
entry). Furthermore most terminals pretending to be xterm don't even send 
the escape sequences the xterm termcap specifies (at least in OpenSuse 10.3 
this even includes the xterm program itself). To patch this all up inputrc, 
vimrc, sysless etc. map like a dozen different escape sequences for keys 
like Home and End. We'll supposedly have to do the same or ssh'ing into 
Haiku from a Linux box will have broken key mappings.
