Hi,

Fredrik Ekdahl wrote:
> Another slightly related question, where/how should S/PDIF pass-through 
> (for AC-3 and DTS) be implemented to be supported on a "system" level? A 
> separate media plugin that encapsulates the stream correctly, or some 
> special case in the media server?

There are some beginnings in the OSS Media Node to offer outputs which 
describe encoded audio formats, depending on what the OSS layer reports as 
device capabilities. Like I said, this code is unfinshed and largely 
untested.

The MediaPlayer would need to check if there is an output on the audio node 
which matches the encoding of the given stream. Then it would need to 
bypass the system mixer and connect the audio directly with that encoded 
output. It's quite a bit of work to handle all the thinkable cases. I guess 
the user would want to configure this behavior, although I had plans to 
make the output nodes configurable anyways. It has become easier to support 
this at all after the rewrite to use Media Nodes directly.

It's certainly a feature not only you want to have, but there are more 
immediate things to do, like supporting h.264 and many other codecs at all. 
The AVI extractor still has problems and we don't even support mpg and mpg2 
video. Since we are so few developers, I am wondering if we shouldn't 
implement more stuff with an ffmpeg backend, like AVI and other extractors. 
I think the code of Marcus' libopendml is much nicer, but it doesn't help 
if nobody has time to keep working on it. Obviously, writing more bridges 
to ffmpeg in the first place is also much work. Puh.

Best regards,
-Stephan
