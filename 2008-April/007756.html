Axel Dörfler wrote:
>> -	B_CPU_INTEL_x86						= 0x1000,
>> -	B_CPU_INTEL_PENTIUM					= 0x1051,
>> +	B_CPU_INTEL_x86						= 0x100000,
>> +	B_CPU_INTEL_PENTIUM					= 0x100501,
> 
> Was there a compelling reason to change the encoding of these 
> constants?

Yes. Intel changed its specs to take into account extended model/family 
for processors and this made the IDs for the processors change.

> I mean it's not really that bad, as most apps shouldn't care about 
> them, and most BeOS app won't be able to show the correct (modern) CPU 
> anyway, and we deliver most of them that do (like ProcessController).
> But still, this breaks compatibility, so we might want to find a better 
> solution.

Well, this is the correct solution and based on the points you 
mentioned, it is a harmless change. We could special case all CPUs 
before a specific version in the code but this would be a hack. 
Specially because if you do not take the extend family/model into 
consideration, with the old IDs my processor was identified as a Rise 
CPU instead of an Intel one. This would probably be true with any new 
Intel CPU.

-Bruno
