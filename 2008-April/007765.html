>
> I just thought it seemed like a simpler solution since otherwise every
> pref has to have special case code for read-only, which seems wasteful
> if it's only ever going to be used for the LiveCD anyways, which in
> theory would only be used once for a little poking around at which
> point the user would likely install to disk or pitch it anyways.

I wasn't thinking of a special case code but replacing with a 
simpler/cleaner code that doesn't have to know if it's running on a read 
only system.
The preflet would communicate with the input server via BMessages as 
suggested by Axel, and the input server would take care of the 
persistency question.

Besides the live cd question, there are other arguments in favor of this 
solution:
Avoid code duplication: The keymap preflet duplicates file io code from 
the input server.
Allow consistent integration of third party apps/preflets (ex: deskbar 
keymap switching applet) .
