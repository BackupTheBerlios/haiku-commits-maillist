On 2008-04-25 at 10:19:33 [+0200], François Revol <revol at free.fr> wrote:
> > * Publish /dev/ptmx. Opening it will open a fresh pty master. In
> >   principle /dev/pt/ is obsolete now, but I guess we keep it around
> > for
> >   compatibility with BeOS. Though there shouldn't be many
> > applications
> >   opening a pty, that we might be interested in...
> 
> I do have several ports around where I wrote a getpty() that used those
> paths

If they are ports it shouldn't be too problematic to update them. If the 
packages try to be portable they will probably have support for 
posix_openpt() or openpty() anyway, which will allow to remove the 
work-around altogether.

> (besides openssh which we can fix).

OpenSSH uses openpty() if available.

> Also, I'd rather keep usint those paths instead as they would make
> things clearer in lsof when we have it. Having all teams with an fd to
> /dev/ptmx doesn't really help finding which is which. Unless we can
> hack the inode back to the other path.

"All teams" would be those that open a pty master only. And unless you're 
just debugging such a program (in which case there's likely only one 
instance), you will usually be interested in slave only, anyway.

Anyway /dev/ptmx is undeniably a more elegant solution than have 64 
/dev/pt/ entries around all the time.
