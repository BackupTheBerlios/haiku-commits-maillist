On 2008-04-14 at 18:54:59 [+0200], Rene Gollent <anevilyak at gmail.com> wrote:
> On Mon, Apr 14, 2008 at 11:49 AM, Ingo Weinhold <ingo_weinhold at gmx.de> 
> wrote:
> >  Although it probably still needs a bit of work, my recent changes to the
> >  file system interface already laid the groundwork for adding layers on 
> >  top
> >  of mounted file system. So it should be possible to implement a layer 
> >  that
> >  adds write support to a read-only volume by storing the changes in 
> >  memory.
> >  When the kernel mounts the boot volume and finds it read-only, it would
> >  simply put that layer on top of it.
> >
> >  With the same technique an attribute/query support layer could add
> >  attribute/query support to a standard ISO9660 CD (the additional data 
> >  could
> >  be stored in special folders). This would finally allow us to create
> >  bootable single-track Haiku CDs.
> 
> Oh nice, I wasn't sure exactly what all the VFS layer changes were
> intended to cover. Out of curiosity, would it be 'safe' for someone to
> start adapting the other filesystems to the new model at this point,
> or do you still have breaking changes planned?

Not planned, no, but the provisions for FS layering are completely unused 
ATM and might not yet meet the requirements. I hope that only small 
adjustments will be needed, but I won't know until implementing such a layer 
(I was thinking of the attribute/query support, BTW). If anyone needs/wants 
a particular FS to work right now, feel free to adjust it to the current 
interface -- that would be much appreciated, even. It's just that *I* would 
rather want to wait with doing it. I can give an overview of what has 
changed and how FSs would need to be adjusted, if desired.

CU, Ingo
