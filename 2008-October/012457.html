On 2008-10-07 at 23:44:55 [+0200], Stephan Assmus <superstippi at gmx.de> wrote:
> 
> bonefish at BerliOS wrote:
> >   This optimization speeds up a "hello world" compilation about 20% on my
> >   machine (KDEBUG turned off, freshly booted), but interestingly it has
> >   virtually no effect on the "-j2" haiku build time.
> 
> Pretty cool! Maybe the haiku build time is primarily influenced by the I/O?
> In another words, the implementation of pre-caching could be most helpful?

At least in Haiku building Haiku doesn't seem to be I/O bound -- the HD 
light is far from flashing permanently. In Linux and FreeBSD it's 
significantly more (maybe 50% of the time), but they also build about 5 
times faster than Haiku. So as I see it ATM Haiku's main performance 
problems likely aren't I/O related.

> I thought this would be so much easier to add with the new I/O-scheduler?

The I/O scheduler makes asynchronous I/O possible, which allows to do some 
stuff that couldn't be done before, but aside from that it doesn't really 
simplify anything.
