On 2009-09-26 at 05:52:31 [+0800], Rene Gollent <anevilyak at gmail.com> wrote:
> On Fri, Sep 25, 2009 at 4:20 PM, Siarzhuk Zharski <zharik at gmx.li> wrote:
> > I found it a bit strange, that gcc4 build requires this header, but gcc2
> > can live without it. The mentioned gcc4 build problem is a linker error
> > about undefined "operator new". May be I have missed something but how 
> > this
> > operator was found by gcc2 linker without this include? :-|
> 
> That's due to a difference in what gcc provides. In gcc4 libstdc++ and
> libsupc++ are part of the compiler distribution. On gcc2 this wasn't
> the case, and operator new and friends were in fact part of libroot.
> Hence why gcc4 needs libsupc++ while gcc2 does not.

More correctly: With gcc2 the C++ runtime was part of libgcc, against which 
we already link all kernel add-ons (cf. HAIKU_KERNEL_ADDON_END_GLUE_CODE), 
since it also contains the C runtime. With gcc4 (since gcc3 even, I 
believe) the C++ runtime no longer lives in libgcc, but in libsupc++ (it 
actually is part of the compiler distribution), which we don't link against 
by default. IMHO we shouldn't use the kernel_cpp.h hack for kernel add-ons, 
but rather link against the static libsupc++. The build system could even 
do that automatically, but since Oliver heroically added the missing 
libsupc++ instances manually, my motivation to implement that isn't that 
big.

CU, Ingo
