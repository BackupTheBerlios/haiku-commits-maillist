Stephan Assmus <superstippi at gmx.de> wrote:
> > IMO, for codecs, we should always use existing implementations 
> > where 
> > possible due to their complexity alone. For (de)muxer, I would 
> > usually 
> > prefer using an official library if such exists (for maximum 
> > compatibility), if the license fits, or even writing it ourselves 
> > if 
> > someone enjoys it :-)
> Agreed. As long as it has equal quality and it is maintained.

At least in the long run :-)

> > Other than that, I think it's always up to the future maintainer of 
> > the 
> > code to decide whether or not to keep/replace a certain piece of 
> > software. If Stephan is not interested in maintaining our ASF 
> > demuxer, 
> > why shouldn't he use the one of ffmpeg? But if we have a maintainer 
> > for 
> > it, who makes sure bugs are fixed, why should we use ffmpeg for 
> > that?
> Since the maintainer is working on many different things anyway, he 
> can use 
> the free time to work on something else, which cannot be so easily 
> fixed by 
> re-using existing code.

But that's completely up to the maintainer to decide.

> > Using VLC vs. our own MediaPlayer is actually a different problem, 
> > because all applications using the Media Kit will benefit from good 
> > and 
> > many codecs; the work we bury there pays off at many fronts.
> That wouldn't even be my argument. I think there is simply a 
> different 
> level of complexity. A media player is sufficiently complex that it 
> interests me how it works and how I can use it. At a certain level, 
> it's 
> just about watching a movie, but anyone who has used a media player 
> knows 
> that there is more to it and the experience can be better or worse. 
> At the 
> level of the whole operating system, it is even more the case, and 
> there is 
> so many details that influence this, that helping to create Haiku is 
> totally worth it to me. However, demuxing a video file and decoding a 
> stream is something very, very low-level. Comparing that to Haiku is 
> like 
> comparing stones to houses.

I think it's always good to reuse existing code if a) the license fits, 
b) the quality/maintainability is good and meets our goals (one might 
need to fix bugs in that code, too), and c) the imported code is not a 
bloated beast, but does exactly what we need. IMO ffmpeg well fulfills 
all of that (if GPL is turned off).

The only problem I see with ffmpeg is the lack of releases, and the 
therefore uncertain quality of whatever revision we use.

Bye,
   Axel.
