*shrug*

I personally don't mind recompiling stuff as necessary and providing
updated packages for those who want to use GCC4 Haiku, if the source
API changes, the sources for popular apps need to be updated anyway,
so we might as well do it as we go if possible.

In this case, Artur provided the package, I tested it for him (and had
him fix a few things), so it seems reasonable to include it :)

Rene doesn't seem to have a problem maintaining GCC4 Vision, GCC4
Firefox on the other hand is sort of a large project.

>  In the case of Pe, there may be the benefit of faster launch times, since
>  most libs will be already loaded (unlike if it's a GCC2 package on a GCC4
>  Haiku). I am just saying the benefits need to outweight the additional work
>  to maintain and test these packages.

In the case of Pe, it's very useful for anyone who wishes to work on a
GCC4-only Haiku and do development - the number of portable software
packages out there that require gcc3 or newer is increasing all the
time, and so if we want to embrace these, we need to start seriously
considering supporting GCC4 Haiku.

all IMHO anyway, but for me GCC2 Haiku or hybrid builds is very
quickly losing my interest personally.
